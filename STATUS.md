# Kill-experiment verdict — run 2026-07-10 → 2026-07-11

## VERDICT: KEEP (pre-registered rule met, decisively)

**N = 35.** Of 48 depth≥6 tasks solved by the bootstrap loop, **35 cannot be
solved by `timeout 120 ./enumerate <task> -d 8 --no-branch`** (all 35 timeout,
none returned failure). The pre-registered rule — KILL if N = 0 — is answered:
model-guided search with exhaustive leaf completion (R=3) reaches programs a
120s brute force cannot. Full detail: `kill_report/crosscheck.json`.

**A/B sanity gate: PASS** (`kill_report/ab_summary.json`). Same round-0
checkpoint, same 500-expansion budget, on the 500 original deep tasks:
R=0 solved 392 with nothing past depth 6; R=3 solved 397 including 2×d7 + 4×d8.
The completion mechanism pays for itself at equal budget — the deep solves are
not brute force wearing a trench coat (the R=0 arm had the same brute-force-free
model and got none of them).

### The 35, decomposed

- **15 are loop-generated frontier tasks** (`syn_1inp_len10_*`, kernel length
  10, created by the adaptive task generator at the round-2/3 boundary). The
  loop's solutions (d7-d8) are *shorter* than the generating kernels — real
  deep synthesis, solved by the round-2 model + R=3 completion during round-3's
  search phase (2026-07-10 22:01 onward, per record mtimes).
- **20 are original deep-set tasks** (len5-len8 kernels) solved at d6-d8. For
  the len5/len6 ones, the loop's recorded depth may exceed the (unknown, shorter)
  optimum — the claim is brute-force *unreachability in 120s*, not optimality
  of the loop's solution. The registered rule counts by recorded depth
  (`rec[22]+rec[23]`), exactly as operationalized in `run_kill_experiment.sh`
  step 7.

### Deviations from the registered protocol (all wall-clock, none verdict-threatening)

1. **Truncated mid-round-3 of 10.** Round search phases cost 2h → 4h → 23h+
   (incomplete): exhaustive completion calls average 20-31 s (measured:
   R1 56 calls / 23.7B states / 1272s; R2 328 calls / 174.5B states / 10,137s)
   versus the 10-150ms/node estimate in issue #7. Protocol completion was weeks
   away on an 8-core box. Truncation can only *understate* N — more rounds add
   solves, never subtract — and the A/B control was complete before truncation.
2. **Cross-check parallelized 6-way** on an otherwise idle box (per-task
   semantics identical to step 7: same command, same 120s timeout per task).
3. Killed during the post-search round-3 retrain; round-3 search stats are
   partially flushed in `logs/loop.log` (on-box only; logs contain absolute
   paths and stay out of the public repo).

### Epistemic caveats, labeled

- Loop solutions are verified by `bootstrap.py`'s internal test replay
  (unit-tested, `test_exhaustive.py` 9/9); they were **not independently
  re-executed** for this report. The states files store labels, not programs.
- 2 of the 48 cross-checked tasks had their JSONs regenerated by the frontier
  generator mid-run; both were checked for consistency — the one that landed
  BEYOND (`syn_1inp_len8_000088`) was solved *after* its spec changed, and the
  cross-check tested the same current spec. No contamination.
- Val F1 declines across rounds (0.069 → 0.036) are **not comparable** — each
  round's val split contains newly generated, harder frontier tasks. Early
  reads of this as "loop collapse" were wrong; the loop was quietly solving
  the frontier while its val metric sank.

## Decision tree position (from the pre-registered tree below)

N ≥ 1 → KEEP. **Strong-KEEP is NOT cleanly met**: new-solutions were monotone
through round 2 (397 → 425, round 3 partial), but "advanced ≥2 depths past the
brute-force-solvable set" is ambiguous here — the 120s brute force solves *some*
d7-d8 tasks (the fast ones), so the advance is "beyond reach within depths 6-8,"
not "+2 whole depths." Per the registered tree this is the **marginal KEEP**
arm: *one more loop run at MAX_ROUNDS=20 before any new code.*

**Practical tension, flagged for the owner:** a MAX_ROUNDS=20 run at current
completion costs is weeks of wall-clock. The actual bottleneck is the 20-30s
completion call (sublanguage restriction and/or threading the C completion are
the obvious fixes — but that is "new code" the tree says to defer). The
strategic context also matters: the AlphaDev/libc++ analysis in README.md was
written *before* this verdict and its salience argument is unchanged by KEEP —
the machinery works; whether the target is worth more investment is a strategy
call, not a science call. **Decision deferred to the owner** (session task #2);
nothing is pre-authorized.

# Pre-registered kill rule (registered 2026-07-07, before any run — kept verbatim for provenance)

No numeric threshold existed anywhere in the repo or issues; the closest
verbatim commitment is issue #6's key metric: *"Can the model solve
depth-5+ tasks that neither random search nor brute force (120s timeout)
can reach?"* The registered rule (also in `run_kill_experiment.sh` header):

**KILL** if, after the loop with `--exhaustive-radius 3` runs to
escalation exhaustion or 10 rounds, the solved frontier contains **zero
depth>=6 tasks** that `timeout 120 ./enumerate <task> -d 8 --no-branch`
cannot also solve. Depth 6 rather than issue #6's depth 5 because R=3
completion hands the model depth 5-6 nearly for free (the model navigates
only depth 2-3), so depth 5 no longer tests the model. `--no-branch` so
the brute-force baseline searches the same branchless space as the model
(branch mode explores a bigger space and would time out more easily,
biasing toward a false KEEP).

## Threshold decision tree (as registered)

`kill_report/crosscheck.json` → `beyond_brute_force` = N:

- **N == 0 → KILLED.** Write the tombstone: leaf completion could not turn
  a depth-4-frontier model into a beyond-brute-force solver. Keep
  `kill_report/`, `logs/loop.log`, and the A/B summary as calibration
  artifacts; the branch dies with a clean conscience. Do not iterate on
  R=4, bigger models, or more rounds — those are exactly the escalations
  the loop already burned through before stopping.
- **N >= 1 and frontier advanced >=2 depths past the brute-force-solvable
  set with monotone new-solutions per round (`logs/loop.log` round
  summaries) → strong KEEP / SURPRISE.** Next capped investment (max ~1
  agent-day): raise search `max_depth` past the hardcoded 8 (issue #7
  targets 12-15+), rerun the loop, and re-verify the depth-7-8 claim from
  issue #7 that this experiment's R=0 A/B arm will have tested.
- **N >= 1 but weaker (no 2-depth advance or non-monotone) → marginal
  KEEP.** One more loop run at `MAX_ROUNDS=20` before any new code.
- **A/B sanity gate (`kill_report/ab_summary.json`):** if R=3 does not
  beat R=0 on solve rate at equal budget, the completion isn't paying for
  itself — treat any N >= 1 as suspect and inspect whether the "wins" came
  from brute force alone (model prefix depth <= 2).
